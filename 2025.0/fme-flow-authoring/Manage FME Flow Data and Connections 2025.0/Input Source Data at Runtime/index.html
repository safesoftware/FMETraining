In general, it's easy to author a workspace for FME Flow because what works on FME Form can be published and run on FME Flow with minimal alteration of the source data.

However, there are additional methods that can be used to take source data management on FME Flow to the next level. These methods include:

  • Managing file datasets.
    • Publishing file datasets to a repository or Resources.
    • Uploading file datasets at run time.
    • Storing file datasets on the Resources file system.
  • Using Database Connections for data stored in databases
  • Using Web Connections for data stored on a web service.

This course will work through the methods of managing source data with FME Flow. However, if you want to learn more about data integration in general, we recommend you browse these learning resources:

  • What is Data Integration?

Source Data Published Parameters

When workspaces are published to FME Flow, the Reader parameter values for source data are maintained. The parameter values are only altered if they are Published Parameters, created as User Parameters in FME Workbench. Published Parameters that take source data as input to a workspace are useful in self-serve workflows that gather user input. 

If a default value is provided to the user parameter, it will be displayed in the Published Parameters on FME Flow when running the workspace. You can keep the default value or supply a different source dataset. 

Input Source Data Files

Providing source data files as parameters allows for ultimate flexibility in self-serve workflows. Look through the different options to input source data into your workspace on FME Flow.

You can enter URLs for datasets hosted online (e.g., a public dataset, an open data portal, or an internal server). When you enter the URL and run the workspace, FME Flow downloads the data directly from the online source. This saves you from downloading it from the online source and then uploading it to FME Flow, and is especially useful for large files. Data from live links can also be processed by automation to ensure the most up-to-date data is always used for your workflows. 

You can enter direct file paths to access datasets on the FME Flow server's local file system. You can enter file paths like C:\Data\zoning.shp or network share paths like \\fileserver\gis\roads.tab. This method is useful for server-side file access without uploading anything through the browser. FME Flow will read the data from the path instead of uploading the dataset through the Browse File System option. 

This option only works if FME Flow has permission to access the file path provided. Unless FME Flow is installed on your computer, it doesn't have access to your C: drive and will access the C: drive on the server it is installed on. For shared network drives, the FME Flow service account must have permission to access the file and it's folder location.  See Workspace Fails on FME Flow When Using UNC Paths for more information. 

You will want to use the Browse Resources option when the source file is already uploaded to FME Flow's Resources, such as in the Data or Temp folders. It's ideal to store data in Resources if the file will be used across multiple jobs or workflows to avoid repeated uploads, ensure consistency, and support shared access. 

You can also browse your file system to select the file to upload to FME Flow. This option is practical when you prefer to navigate your folders manually instead of drag-and-drop, or if your device does not support drag-and-drop. Uploading a file to FME Flow this way is better than using the Enter File Path option if FME Flow is not hosted on the machine you're using to access FME Flow to ensure access to the file. 

The selected file will appear as Uploaded for FME Flow to use as source data. 

The drag-and-drop file method is best if you have a local file ready and want a quick and easy upload. It is also very convenient if you already have the desired file open in the file explorer. Ultimately, this option is best for ad-hoc one-time runs of your workspace where the file is only needed one time. 

Once you drop the file, it will upload and be ready to use as input to your workspace. 

Temporary Uploads

If you add your source datasets by uploading at runtime, they will be stored in a temporary folder on FME Flow. The folder for a workspace's temporary uploads is found in Resources > System > temp > upload > Repository > Workspace > User. By default, the files are kept in this folder for 24 hours and removed by system clean-up tasks. 

Both the Browse File System and Drop Files input options from above store the data as a temporary upload. 

Suppose you rerun the workspace before the clean-up tasks remove the temporary data. In that case, you can select previously uploaded files for the workspace by clicking the Browse Resources option in the source file parameter and navigating to the Temporary Uploads tab. You may select the file as the source dataset to run the workspace again. 

There are some cautions and limitations to consider when data is uploaded for translation by the end-user:

  • Giving the user upload ability is risky because their dataset's schema has to match the workspace's schema definition; otherwise, the translation will fail with unexpected input. Alternatively, a dynamic (and maybe generic) translation could be used to avoid such issues.
  • Data uploaded by the user is only temporarily available. The System Cleanup page shows us that such files are (by default) deleted when they become more than 24 hours old. User uploads are not a long-term solution.
  • In theory, data uploaded by the user is accessible through Resources > System > temp > upload. However, it's not a commonly accessed location where an end-user would be expected to find it. Therefore, temporary data should be considered inaccessible by any other means and unavailable for use by any other workspace.

Exercise

Sven, a city government planning analyst, is starting to run his workflows on FME Flow. He created a workspace that clips election voting location data by neighborhood in Vancouver and writes the data in a geodatabase. While the neighborhood boundaries data does not change, the voting location data may change frequently, so that the user can upload new temporary data through a Published Parameter. 

Follow along with Sven as he continues his workflow by publishing the workspace to FME Flow and then running it on FME Flow with temporary data. 

1) Open Starting Workspace

Sven opens his workspace (C:\FMEData\Workspaces\DeployWorkflowsWithFMEFlow\input-source-data-at-runtime.fmw) in FME Workbench. It reads in Voting Places (GML) and Neighborhoods (KML) data, clips the voting places to the neighborhoods, and then writes the data to a geodatabase to keep it updated with the most accurate voting information. 

2) Add Source Data Parameter

To take user input for the VotingPlaces data, Sven must use a parameter to accept the data on FME Flow. To do so, Sven expands the ElectionVoting reader settings in the Navigator, right-clicks the Source Geography Markup Language (GML) File(s) parameter, and selects Create User Parameter. 

He does not need to alter any default settings for the parameter, so he clicks OK and then makes sure he sees his new parameter under User Parameters. 

3) Publish to FME Flow

Sven publishes the workspace to FME Flow, saves it in the Training repository, and registers it with the Job Submitter service. 

Expand for Steps on Publishing to FME Flow
The publishing wizard sets all configurations for publishing workspaces to FME Flow. To open it, click Publish in the top toolbar.
The first steps are to confirm your FME Flow connection, select a Repository to house the workspace, and optionally alter the workspace's name. To create a new Repository, click New and enter its details.  
Optionally, you may Upload data files to publish the workspace's source data files to FME Flow along with the workspace. You can control which files from the Select Files... option.
Next, if your workspace has any connections, you may select whether or not they will also be published to FME Flow. 
The last step of publishing is to register services for the workspace to use on FME Flow. 
After publishing, the translation log generates a report on the publishing and provides a direct link to run the workspace on FME Flow.
For more information on the FME Flow Services and Publishing to FME Flow, see Deploy Workflows on FME Flow. ** add link **

4) Open Run Workspace on FME Flow

Sven navigates to FME Flow and opens the Run Workspace page and configures it to run his recently published workspace. 

5) Configure Source Data and Run Workspace

Under Published Parameters, Sven removes the default file path value from the Source Geography Markup Language (GML) File(s) parameter. 

Sven knows that GML files need accompanying XSD (XML Schema Definition) files to be correctly interpreted. This means he needs to input the GML file as the source to the workspace and have the supporting XSD file in the same directory. To upload both files at runtime, Sven clicks Browse Resources and goes to the Temporary Uploads tab. He drags both ElectionVoting.gml and ElectionVoting.xsd (C:\FMEData\Data\Elections\) files to the Upload Files area. Because he only needs the ElectionVoting.gml file as the source file, he deselects the XSD file in the Selected Items column. 

Sven clicks OK, confirms that only the GML file is included for the source parameter, and clicks Run. 

This method is useful for formats that require "sidecar" files, such as Esri Shapefile or MapInfo TAB. If the XSD file was not required, Sven could've simply uploaded the GML file directly on the Run Workspace page. 

If your workspace fails here, make sure that you've uploaded both the ElectionVoting.gml and ElectionVoting.xsd files (C:\FMEData\Data\Elections\) to Temporary Uploads, but only the ElectionVoting.gml file is selected for the parameter input. 

7) Optional: Re-run Workspace with New Data

Sven's workspace completes successfully. However, he wants to update the geodatabase with new voting location data. So, Sven clicks Run Again and returns to the Run Workspace page. 

The NewElectionVoting.gml file uses the same XML Schema Definition as the previous GML data, and it's still present in Temporary Uploads for 24 hours by default. So, Sven drops the NewElectionVoting.gml file (C:\FMEData\Data\Elections\NewElectionVoting.gml) in the Upload Files area on the Run Workspace page and removes the ElectionVoting.gml file. 

Sven reruns the workspace to overwrite the geodatabase with the new data. 

8) Optional: View Temporary Data in Resources

To review his temporarily uploaded files, Sven navigates to Resources > System > temp > upload > Training > input-source-data-at-runtime.fmw > admin.
